## 3. 业务流程影响

### 3.1 动物详情页 (Animal Details)
*   **移除**: 请移除原有的“总体筹款进度条”和“筹款目标金额”显示。
*   **新增**: 请渲染一个**活动列表** (Campaign List)，遍历 `animal.campaigns` 数据。
    *   每个活动卡片应显示：标题、进度 (Current/Target)、状态标签。
    *   提供“捐款”按钮，点击后调用捐款接口，传入该活动的 `id`。

*   **证明提交**: 在任务详情页，增加“上传证明”的功能，调用 `health-proof` 接口。

    *   *错误做法*: "捐给 Buddy 50元" (缺少 CampaignId)

## 8. 认证与安全 (Auth & Security)

### 8.1 发送验证码 (Send OTP)
**接口**: `POST /api/auth/send-verification`
**参数**:
```json
{
  "email": "user@example.com",
  "type": 0  // [可选] 0 = Email Verification (默认), 1 = Password Reset
}
```
**描述**: 向指定邮箱发送 6 位验证码。有效期 10 分钟。
*   `type=0`: 用于注册/登录验证。会检查是否已验证。
*   `type=1`: 用于忘记密码。不会检查是否已验证。

### 8.2 验证邮箱 (Verify OTP)
**接口**: `POST /api/auth/verify-email`
**参数**:
```json
{
  "email": "user@example.com",
  "code": "123456"
}
```
**描述**: 验证用户输入的 OTP。验证成功后，用户账号会被激活 (`IsApproved=true`)，从而解锁登录限制。

### 8.3 登录接口行为变更
**接口**: `POST /api/auth/login`
**变更**: 如果用户未通过 OTP 验证（即 `IsApproved=false`），登录接口目前可能会返回：
*   **403 Forbidden**: `{"message": "Account is pending approval."}`
*   或者允许登录但角色受限（取决于具体实现，请以测试为准）。一般建议前端在收到 403 时引导用户去验证邮箱。

---

### 8.4 忘记密码 (Forgot Password)
**接口**: `POST /api/auth/forgot-password`
**参数**:
```json
{
  "email": "user@example.com"
}
```
**描述**: 发送 6 位验证码到邮箱（用于重置密码）。
*   **注意**: 现已改为发送 OTP。前端也可以直接调用 `send-verification` 接口并传 `type=1` 来替代此接口。

### 8.5 重置密码 (Reset Password)
**接口**: `POST /api/auth/reset-password`
**参数**:
```json
{
  "email": "user@example.com",
  "code": "123456",           // [变更] 原 resetToken 改为 code
  "newPassword": "NewPassword123!"
}
```
**描述**: 使用收到的验证码重置密码。系统会校验验证码的有效性（Type=1）。
**安全限制**: 新密码不能与当前密码相同。如果相同，将返回错误：`"New password cannot be the same as your current password."`

## 9. 前端集成指南：用户注册与验证流程 (Register & Verify Flow)

为了确保用户邮箱的真实性，我们引入了强制的 OTP (One-Time Password) 验证流程。以下是前端对接的完整步骤：

### 流程概览
1.  **用户注册**: 调用注册接口。
2.  **阻止登录**: 用户尝试登录，系统返回 `403` (未验证)。
3.  **发送验证码**: 前端调用发送接口，向用户邮箱发送 OTP。
4.  **验证邮箱**: 用户输入 OTP，前端调用验证接口。
5.  **登录成功**: 验证通过后，用户再次登录成功。

### 详细步骤与接口参数

#### 步骤 1: 用户注册 (User Register)
*   **接口**: `POST /api/auth/register`
*   **输入**:
    ```json
    {
      "username": "newuser",
      "email": "user@example.com",
      "password": "Password123!",
      "role": "User",
      "fullName": "New User"
    }
    ```
*   **预期返回**: `200 OK`
*   **Message**: "Registration successful. Please verify your email via OTP to login."
*   **状态**: 此时数据库中 `User.IsApproved = false`。

#### 步骤 2: 尝试登录 (Login Attempt)
*   **接口**: `POST /api/auth/login`
*   **输入**:
    ```json
    {
      "username": "newuser",
      "password": "Password123!"
    }
    ```
*   **预期返回**:
    *   **成功 (200)**: 如果是 `Volunteer` 角色，允许登录但权限可能受限。
    *   **失败 (400/403)**:
        *   **普通用户**: "Account is pending verification." (需验证邮箱)
        *   **Shelter (场景1)**: "Please verify your email first." (需验证邮箱)
        *   **Shelter (场景2)**: "Email verified. Waiting for Admin approval." (邮箱已验，等管理员批)
*   **前端处理**: 
    *   收到 "verification" 相关错误 -> 跳转 OTP 页面。
    *   收到 "Waiting for Admin approval" -> 提示“您的申请已提交，请耐心等待管理员审核”。

#### 步骤 3: 发送验证码 (Send OTP)
*   **接口**: `POST /api/auth/send-verification`
*   **输入**:
    ```json
    {
      "email": "user@example.com"
    }
    ```
*   **预期返回**: `200 OK` `{"message": "Verification email sent."}`
*   **系统行为**: 生成 6 位数字验证码发送到该邮箱，有效期 10 分钟。

#### 步骤 4: 提交验证 (Verify OTP)
*   **接口**: `POST /api/auth/verify-email`
*   **输入**:
    ```json
    {
      "email": "user@example.com",
      "code": "123456" // 用户收到的6位验证码
    }
    ```
*   **预期返回**: `200 OK` `{"message": "Email verified successfully."}`
*   **系统行为**: 验证成功后，将用户的 `IsApproved` 设置为 `true`。

#### 步骤 5: 再次登录 (Login Success)
*   用户再次调用 `POST /api/auth/login`。
*   **预期返回**: `200 OK`，包含 JWT Token。

### 11.3 业务规则变更 (Business Rules)

#### A. 捐赠限制 (Donation Limit)
- **规则**: 捐款金额不能超过活动剩余所需金额 (`Target - Current`)。
- **错误**: 若超额，返回 500 错误 (这里建议后续优化为 400)，Message: `"Donation exceeds required amount. Maximum allowed..."`。
- **规则**: 若活动已筹满，由系统自动关闭捐赠通道，再次尝试捐赠会报错 `"Campaign is already fully funded."`。

#### B. 紧急募资 (Emergency Campaign)
- **限制**: 仅允许为流浪状态 (`Status="Stray"`) 的动物发起紧急募资。若为已领养动物，接口将报错。

#### C. 收容接收 (Shelter Intake)
- **限制**: 如果流浪动物有状态为 `Pending` 的领养申请，收容所无法执行接收 (`Intake`) 操作。
- **错误**: `"This animal has pending adoption applications..."`。
